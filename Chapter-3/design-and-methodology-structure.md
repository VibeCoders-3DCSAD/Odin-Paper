### RESEARCH DESIGN

**1. Descriptive Research**

**Purpose:**

Descriptive research in computer science is focused on gathering information, exploring, and providing a detailed
picture of a specific phenomenon without manipulating variables. The goal is to describe the existing state of things,
whether it’s system behavior, software features, or trends in user interactions.

**Explanation for Thesis:**

- **Objective** : Describe the existing state of a particular system, framework, technology, or process.
- **Scope** : Discuss the scope of the data being observed. For example, you might describe the current
    performance of an algorithm, the user interface design of a software tool, or the architecture of a system.
- **Data Collection** : Outline how you’ll collect data (e.g., logs, surveys, usage statistics, system performance
    metrics, etc.).
- **Analysis** : Describe how you will analyze this data (e.g., statistical methods, graphical representations, pattern
    recognition).
- **Application in Computer Science** : This approach is often used when analyzing the current state of software,
    systems, or technologies (e.g., performance analysis of an existing algorithm or comparing the features of
    different programming languages).
**2. Experimental Research**

**Purpose:**

Experimental research involves manipulating one or more variables to determine their effects on a particular outcome.
In computer science, this typically involves testing algorithms, systems, or software designs under controlled conditions
to observe and measure the effects of certain interventions.

**Explanation for Thesis:**

- **Objective** : Investigate cause-and-effect relationships by manipulating specific variables (e.g., testing the
    efficiency of a new algorithm or comparing different system designs).
- **Control and Variables** : Identify the independent variables (e.g., algorithm parameters, system
    configurations) and dependent variables (e.g., performance metrics like time, memory usage, throughput).
- **Methodology** : Discuss how you plan to carry out experiments—this could include test environments,
    datasets, and tools (e.g., using controlled test cases or simulations to observe how different inputs affect
    outputs).
- **Experimental Setup** : Describe how you will set up the experiments, such as the testing environment,
    software, hardware, and tools.
- **Data Collection** : Specify what kind of data you will collect and how it will be measured (e.g., response time,
    throughput, accuracy, error rate).
- **Analysis** : Explain how you will analyze the data, including any statistical tests or performance comparisons
    you’ll conduct.


**3. Developmental Research**

**Purpose:**

Developmental research in computer science focuses on creating, testing, and refining new technologies, tools, or
systems. This design involves a cycle of building and iterative improvements, with the aim of developing new software,
frameworks, or systems that solve specific problems.

**Explanation for Thesis:**

- **Objective** : Develop a new system, application, algorithm, or tool and evaluate its design, functionality, and
    effectiveness.
- **System Design/Development** : Describe the process of designing and building the system or software,
    including the underlying principles and technologies (e.g., programming languages, software architecture,
    frameworks).
- **Prototyping** : Discuss how you will prototype or create a proof of concept (e.g., developing an early version
    of the software or a partial system to test its feasibility).
- **Testing and Evaluation** : Explain how you will test and evaluate the system or tool you have developed. This
    could involve usability testing, performance benchmarking, or functional validation.
- **Iteration** : Detail how feedback will be incorporated to refine and improve the system, algorithm, or tool. This
    iterative process is a key characteristic of developmental research.
- **Outcome** : The ultimate goal is the creation of a product that can either be deployed in real-world applications
    or contribute to the scientific knowledge in computer science.

### FUNCTIONAL TESTING

- **Definition** : Define **functional testing**. This type of testing focuses on verifying that the system behaves as
    expected according to the functional specifications. It checks whether the software performs the required

- **Purpose** : Explain the goal of functional testing: to ensure that each feature works as expected and meets the
    specified requirements.
- **Key Types of Functional Testing** : Discuss the various types of functional tests used, such as:
    o **Unit Testing** : Testing individual components or functions of the software to ensure they work
       correctly.
    o **Integration Testing** : Ensuring that different components or systems interact correctly.
    o **System Testing** : Validating the entire system against functional requirements.
    o **Acceptance Testing** : Verifying if the software meets user requirements and is ready for release.
- **Test Cases and Scenarios** : Describe how you create test cases for functional testing based on the functional

### NON-FUNCTIONAL TESTING

- **Definition** : Define **non-functional testing**. This type of testing verifies non-functional aspects of the system,
    such as performance, security, usability, and scalability. While functional testing ensures the system works
    as expected, non-functional testing assesses how well the system performs under various conditions.
- **Purpose** : Explain that the goal of non-functional testing is to measure the system’s behavior and performance
    in real-world scenarios, ensuring it can handle expected loads and meets user expectations beyond just
    functional correctness.
- **Key Types of Non-Functional Testing** : Discuss the different types of non-functional testing, which include:
    o **Performance Testing** : Evaluating how the system performs under different loads. This can involve
       testing system response times, throughput, and resource usage under normal and peak loads.
    o **Load Testing** : Testing the system under expected peak loads to see how it behaves under stress.
    o **Stress Testing** : Pushing the system beyond its limits to identify breaking points and ensure it can
       recover gracefully.
    o **Scalability Testing** : Verifying how well the system scales when the number of users or data
       increases.
    o **Security Testing** : Ensuring the system is resistant to security threats, such as SQL injection, cross-
       site scripting (XSS), and unauthorized access.
    o **Usability Testing** : Assessing how easy and intuitive the system is for end-users.
    o **Compatibility Testing** : Verifying that the system works across different browsers, devices, and
       operating systems.
- **Test Scenarios and Criteria** : Discuss how non-functional test scenarios are designed, which might involve
    setting performance benchmarks (e.g., maximum response time under load), evaluating security against
    specific attack vectors, or assessing usability with user feedback.
- **Tools and Techniques** : Mention the tools used for non-functional testing, such as **JMeter** , **LoadRunner** ,
    **OWASP ZAP** for security testing, or **Google Lighthouse** for usability and performance assessments.


**Comparing Functional and Non-Functional Testing**

- **Functional vs Non-Functional Testing** : Provide a comparison between functional and non-functional testing
    to highlight the different objectives. Functional testing ensures that the system works correctly, while non-
    functional testing evaluates how well the system performs under various conditions.
- **Example** :
    "While functional testing focuses on verifying that the software performs the required tasks, non-functional
    testing ensures that the system can handle real-world scenarios, such as high traffic, security threats, and
    user interaction. Both types of testing are necessary to ensure a complete, robust system."

**Challenges in Functional and Non-Functional Testing**

- Discuss common challenges you might encounter during both types of testing. For example, functional testing
